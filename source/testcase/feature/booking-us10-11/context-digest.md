## Order of Case

- Group by User Story (US10, US11, ...) with a header row per US
- Within each US, follow lifecycle order:
  1. Entry point / navigation
  2. Field input and validation (each field)
  3. Search and selection flows
  4. Edge cases per field (empty results, no data, limits)
  5. Submit / confirm action
  6. Business logic (overlap, past date, status rules)
  7. Cancel / discard flows — mandatory 3 states: no changes→close, changes→discard, changes→stay
- Positive cases before negative/edge cases within each functional group
- UI navigation first, then data input, then validation, then business logic

## Consolidation Strategy

- **MERGE when:**
  - Entry point + defaults in one test (multi-AC: navigation AC + display AC)
  - Progressive field behavior: counter appears → input blocks → all in one test
  - Identical dismiss behavior: Cancel button and X icon do the same thing → one test covers both
  - Edit + apply as one flow (don't split "open edit" from "save edit")
  - Output is the same across inputs (e.g., search valid name + search valid email → same result list → one test with multiple inputs)
  - Same view variant with minor differences (e.g., calendar Day view and Week view share identical behavior → one test covers both)
- **SPLIT when:**
  - Structurally different views (e.g., Day/Week vs Month when Month has distinct display rules)
  - Different outcomes for the same trigger (overlap detected vs no overlap)
  - Different preconditions (new session vs editing existing)
  - Different components (date picker vs time picker vs note field)
- **Step target:** 5-8 steps per test. If only 2-3 steps → likely should merge with a related test. If >10 steps → consider splitting.

## Multi-AC Mapping

- Prefer multi-AC when workflow naturally spans entry point + field behavior
- Format: `US{N} / AC{N}, AC{N}` — list ALL ACs the test verifies
- Never create a test that only verifies defaults without also verifying the entry point
- A single test covering 2-3 ACs is preferred over 3 single-AC tests when the behaviors are sequential

## Boundary Testing Strategy

- Use **representative categories**, not exhaustive enumeration
- For display thresholds with 3+ tiers: test smallest, middle, largest — skip intermediates unless they have distinct behavior
- For character limits: ONE progressive test (counter appears at threshold → input blocks at limit)
- For input methods: keyboard entry = 1 test, picker interaction = separate test, range limits = separate test
- For duration/height display: pick 3 representative values (short, medium, long) — not every possible duration

## Scope

- Happy path: ~60% — standard flows, field inputs, selections, successful operations
- Edge cases: ~25% — empty search, no data states, past dates, overlapping sessions, long text
- Negative cases: ~10% — missing required fields, out-of-range dates, input blocked at limit
- UI/visual: ~5% — color coding, timezone display, responsive height, horizontal stacking
- Every AC must have at least 1 testcase
- Multi-AC testcases explicitly list all covered ACs in Items for Pivot Table
- Cancel/discard: mandatory 3 states (no changes→close, changes→discard, changes→stay)
- Required patterns checklist: close via X icon, long text wrapping, timezone conversion, custom mode e2e, picker + keyboard input tests

## US10: As a coach, I can book a session for a client in the Workspace via “Book Session” CTA so that I can schedule a session with client

### AC1: Entry Point
- Given: I am on the Calendar page on the Booking feature / OR I double click on a blank space on Everfit Calendar
- When: I click on the "Book a session" CTA button / OR I click a time slot on the calendar
- Then:
  - The system should display a "Book a session" popup.
  - The background calendar should be dimmed.
  - Focus should be trapped inside the modal.
  - The "Book a session" popup should include:
    - Popup name: "Book a session"
    - Trainer (disabled, default to WS's Owner)
    - Client (required)
    - Start Date (required)
    - Start Time (required)
    - Type (optional)
    - Name (required)
    - Duration (required)
    - Location (required)
    - Description (optional)
    - Add note to client (optional)
    - CTA buttons: "Book Session" | "Cancel"
    - Close icon
  - If the popup opens via CTA button, the system should prefill:
    - Start date = Today
    - Start time = 7:00 AM
  - If I click a time slot on the calendar, the system should prefill:
    - If the calendar is on Day view or Week view (Time slot will be 1 hour long on these calendar view mode):
      - Start date = clicked date
      - Start time = start hour of the time slot (e.g., if coach clicks 6 am-7 am, start time will be 6 am)
    - If the calendar is on Month view (Time slot will be day long on these calendar view mode):
      - Start date = clicked date
  - Trainer field should be disabled and default to Workspace's Owner.
  - Location field should default to "In-person".
  - Start time field should default to 7:00 AM.
  - Start date field should default to the current day.
  - Duration field should be auto-populated with 30 min.

  - **Duration field**:

### AC3: Book validation and Conflict handling
- Given: The Book a session modal is open
- When: I click "Book Session"
- Then:
  - The system should validate all required fields.
    - If any required field is left empty, the system should highlight it and display an inline message.
  - If all required fields are inputted:
    - If the end date & time is >= current date & time (compare up to minute), the system should check for "Overlap session".
      - It should check if there is an overlap with a valid session of the trainer in the time period via API call to backend.
      - A valid session for overlap check should be:
        - Has the same trainer (Note: in P1, trainer will always be the Owner of the WS).
        - Is not a "canceled" session.
        - Start time and End time overlaps with the one being created (current_start_time & current_end_time for now) based on these conditions:
          - `start_time <= current_start_time AND end_time <= current_end_time`
          - `start_time <= current_start_time AND end_time >= current_end_time`
          - `start_time >= current_start_time AND end_time <= current_end_time`
          - `start_time >= current_start_time AND end_time >= current_end_time`
      - If there is no overlap with a valid session:
        - The system should save data to the database.
        - Close the popup.
        - Render the new session on the calendar, positioning it correctly based on start time and duration.
        - It should navigate coach to the new created session on the calendar (e.g., if at Week view of Jan 26, 2026 and creating a session on Feb 1, 2026, navigate to week of Feb 1, 2026).
      - If there is an overlap with a valid session:
        - The system should display a confirmation popup with Title: "Session Overlap".
        - If there is at least 1 Everfit session overlap:
          - Description text: "You already have another session during this time. Are you sure you want to book this session anyway?"
        - Two actions: "Cancel", "Book Anyway" (focused).
        - When I click "Cancel" or the "x" icon on the top right corner, the system should:
          - Close the confirmation modal.
          - Return me to the "Book a session" popup.
          - All inputted data should be kept.
        - When I click "Book Anyway", the system should save the session (same as no overlap session behavior).
    - If the session end date & time is in the PAST (end date & time < current date & time (compare up to minute)), the system should NOT check Overlap session BUT display the "Past date confirmation popup".
      - Popup name: "Session Date Is In the Past".
      - Message: "You selected a Date and time that is in the past. This session will be saved as Completed. No client communication will be sent. Are you sure you want to book this session anyway?"
      - Two actions: "Cancel", "Book Anyway" (focused).
      - When I click "Cancel" or the "x" icon on the top right corner, the system should:
        - Close the confirmation modal.
        - Return me to the "Book a session" popup.
        - All inputted data should be kept.
      - When I click "Book Anyway", the system should save the session (same as no overlap session behavior).
  - If the session has been created successfully, the system should:
    - Close the "Book a session" popup.
    - Store data to database in `session` and `booking` table.
    - Display toast message: "The session has been added" "Session has been scheduled.".
    - Display the session on the Coach's calendar in the current calendar view.
  - If the session fails to be created, the system should:
    - Keep the "Book a session" popup open.
    - All coach's data input should be persisted and displayed on the popup.
    - Keep the CTA enable so that coach can try again.
    - Display toast message: "Failed to add a session. Please try again".

### AC3: Session rendering on calendar
- Given: Sessions exist in the selected date range
- Then:
  - Should display sessions which are NOT canceled.
  - **Day View is selected**, the system should:
    - Display sessions in a vertical time-slot layout.
    - **Session color**:
      - If the session is tied to a session type: The session should be displayed in the color based on the selected session type.
      - If the session is a custom one: the session should be displayed in the color selected while the session was being created.
    - Session color should distinguish sessions in the past, upcoming, or ongoing.
      - Apply for both session and all day event.
      - Apply for all session status: Completed, Scheduled, Early canceled, Late canceled.
      - Session in the past: one that has `end_time < current_time`.
      - Current session: one that has `end_time >= current_time`.
      - Distinguish based on the hover state, selected state.
    - **Horizontal Stacking for Overlapping Sessions**:
      - Overlapping sessions must be displayed side by side (horizontally stacked) within the same time range.
      - This applies to: Fully overlapping sessions (same start time and end time), Partially overlapping sessions (different durations but overlapping time ranges).
      - **Sorting Order**:
        - Overlapping sessions must be sorted by `start_time`: the earlier session on the left, the later session on the right.
        - If sessions have the same start time, sorted by `duration`: the longer session on the left, the shorter on the right.
        - If sessions have the same start time and duration, sorted by `created_at`: the older session should be displayed on the left, followed by newer sessions to the right.
        - Given there is vacant slot on the calendar timebox, the sessions should be sorted to the left side.
    - **Dynamic Width Adjustment**: When multiple sessions overlap within the same time slot, the width of each session block must be reduced proportionally to fit within the available horizontal space.
    - **Session Content Display**:
      - Session information must be displayed within the session block.
      - If the content exceeds the available space, it should be truncated.
      - Full details can be viewed via the Session Detail Popover.
    - **Responsive Behavior**:
      - Session display must be responsive and adapt appropriately to different screen sizes.
      - Allow clicking each session to open its popup.
    - Each session block displayed on the calendar should include:
      - The session block should be displayed on the calendar aligned with its Start Time.
      - Session should be displayed on the calendar based on the Coach's timezone (the one logged in to the system viewing the session).
      - Based on the Start Time, the session must begin exactly at the corresponding time slot on the calendar grid (similar to Google Calendar behavior).
      - For sessions <= 15 mins, the system should:
        - The session block height must equal 1/4 of a 1-hour time slot.
        - This is fixed and not further reduced.
        - Display session info in 1 line.
        - Session info includes: Client name, Start time, Session name.
      - For sessions > 15 mins, the system should:
        - The session block height must be calculated as: `session duration / 60 mins`.
          - For example: 20-minute session -> 20/60 = 1/3 of 1-hour slot; 30-minute session -> 1/2 of 1-hour slot; 45-minute session -> 3/4 of 1-hour slot; 90-minute session -> 1.5 hour slots.
        - Display session info in the session block rule:
          - For sessions with duration <= 45 mins:
            - Display session info in 1 line.
            - Session info includes: Client name, Start time, Session name.
          - For sessions with duration > 45 mins AND < 60 mins:
            - Display session info in 2 lines.
            - Line 1: Client name, Start time.
            - Line 2: Session name.
          - For sessions with duration >= 60 mins:
            - Display session info in 3 lines.
            - Line 1: Client name.
            - Line 2: Start time – end time (End time = Start time + Durations).
              - If Start time and End time are NOT in the same day AND same am/pm: Only display am/pm for the end time.
              - If the Start time and End time are in the same day AND same am/pm: Display am/pm for both Start time and End time.
            - Line 3: Session name.
  - **Display canceled session**:
    - **Early canceled session**: Display the session with a dash_line, the Client name should be strike through.
    - **Late canceled session**: Display the session with a dash_line, an "i" icon, the Client name should be strike through.
  - **Week View is selected**, the system should:
    - Display sessions across 7 day columns.
    - Session color: Same as Day view.
    - Stack overlapping sessions within the same time range.
    - Sort sessions by start time ascending.
    - If there are sessions with the same start time, sort them by "created time".
    - Multi-day or all-day sessions should be displayed on the top of the calendar area.
    - Across days sessions should be displayed for both days.
      - For example, a session from 11:00 PM Jan 30 to 1:00 AM Feb 1 should be displayed:
        - On the Jan 30 calendar from 11:00 PM to 12:00 AM.
        - On the Feb 1 calendar from 12:00 AM to 1:00 AM.
      - Display each block of session: Same rule as Day View above.
  - **Month View is selected**, the system should:
    - Display sessions as condensed items within each date cell.
    - Should show session info, includes: Start time, Client name.
    - **Session color**:
      - If the session is tied to a session type: The session should be displayed in the color based on the selected session type.
      - If the session is a custom one: the session should be displayed in the color selected while session was creating.
    - Session color should distinguish the sessions in the past and the upcoming or ongoing sessions.
      - Apply for both session and all day event.
      - Apply for all session status: Completed, Scheduled, Early canceled, Late canceled.
      - Session in the past: one that has `end_time < current_time`.
      - Current session: one that has `end_time >= current_time`.
    - Handle responsive based on user's device screen.
    - If there is a session that has not been displayed due to the screen size, the system should:
      - Display a "[The number of remaining session] more" indicator when sessions exceed the visible limit.
      - When I click "[The number of remaining session] more", the system should:
        - Open a popover listing all sessions for that date (MAX HEIGHT: 292).
        - Allow coach to scroll up and down to view the whole list.
        - Display sessions sorted by start time ascending.
