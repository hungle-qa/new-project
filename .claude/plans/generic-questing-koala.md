# Brainstorm: Remove TESTABLE Section from Context-Digest

> **Domain:** AI Workflow
> **Skills activated:** Prompt Optimization, Token Cost, Agentic Pipeline Reliability

## What You Want

Remove the `## Test Scope > ### TESTABLE` section from `context-digest.md`. This section is a structured list of US/AC titles extracted from the spec (e.g., "US10 - AC1: Entry Point, AC2: Session data input...").

---

## Current State

The TESTABLE section is ~12 lines in the booking digest (lines 525-536). Meanwhile the **full spec** is already embedded above it (lines 147-521, ~375 lines). The TESTABLE section is essentially a table of contents extracted from the spec.

**Generated by:** `buildTestScope()` in `TestcaseDigestHelpers.ts`
**Consumed by:** `write.md`, `write-deep.md`, `update.md`

---

## Risk Analysis

### LOW RISK: Token savings

- TESTABLE is ~12 lines vs ~375 lines of spec. Removing saves **< 3% of digest tokens**. Negligible cost impact.
- Verdict: **Not a meaningful reason to remove.**

### MEDIUM RISK: Write skill coverage drift

`write.md` line 29 says: *"Generate testcases for items in `## Test Scope > ### TESTABLE`"*

Without TESTABLE, the LLM reads the full spec and decides what to test. This **usually works** — the spec is clear about US/AC structure. But:
- If the spec contains non-testable sections (permissions, business rules, notes), the LLM might generate tests for those too
- The explicit TESTABLE list acts as a **whitelist filter** — "test THESE, ignore everything else"
- Risk level depends on spec quality. Clean specs = low risk. Messy specs with lots of context = higher risk.

### HIGH RISK: Write-deep scaffold validation breaks

`write-deep.md` Step 4 (Validate Scaffold) says:

> **AC Coverage:** Every AC in `## Test Scope > ### TESTABLE` has at least 1 test

This is a **hard validation gate**. The LLM checks the scaffold against the TESTABLE list to ensure 100% AC coverage. Without it:
- The LLM has no explicit checklist to validate against
- It must infer testable ACs from the full spec — possible but less deterministic
- Coverage gaps become harder to catch programmatically

### MEDIUM RISK: Future OUT OF SCOPE feature blocked

`buildTestScope()` already calculates `outOfScope` (knowledge topics not in spec) but doesn't write it to the digest. If you later want to add OUT OF SCOPE filtering, removing TESTABLE removes the infrastructure that supports it.

---

## Devil's Advocate

**"The spec already has all the info — TESTABLE is redundant"**

Technically true, but TESTABLE serves a different purpose than the spec. The spec describes WHAT features do. TESTABLE tells the LLM WHERE to aim. It's the difference between giving someone a textbook vs giving them a textbook + a study guide with "chapters on the exam." Both have the same info, but the study guide prevents wasted effort on non-exam chapters.

**The real question:** Do you trust the LLM to extract the right coverage targets from the raw spec every time? If yes → remove. If no → keep the guardrail.

---

## Recommendation

**Keep TESTABLE but consider these alternatives:**

| Option | Effort | Risk | Token Save |
|--------|--------|------|------------|
| A. Keep as-is | None | None | 0% |
| B. Remove TESTABLE, update skill instructions to "cover all US/AC in spec" | Small | Medium — coverage drift possible | ~3% |
| C. Inline TESTABLE into spec headers (add `[TESTABLE]` markers) | Medium | Low — same guardrail, no separate section | ~2% |
| D. Move TESTABLE to a comment/meta block the LLM sees but is smaller | Small | Low | ~1% |

**My pick: Option A (keep).** The token savings don't justify the reliability risk. The TESTABLE section is 12 lines — it costs almost nothing and provides a hard coverage contract that write-deep's validation depends on.

If you're motivated by **digest simplification** rather than token savings, Option C (inline markers) gives you a cleaner digest without losing the guardrail.
