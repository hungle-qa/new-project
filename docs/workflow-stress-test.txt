WORKFLOW STRESS TEST REPORT
===========================

1. ORTHOGONALITY — Redundancy Risk: HIGH

Strategy vs Structure collision. Both compete for hierarchy control. spec-driven hardcodes "US=Level 1, AC=Level 2." Structure tree in config.md defines its own tree. write.md resolves this with "Structure defined → STRICTLY follow tree" — but that silently neuters the strategy's mapping rules. If a user picks spec-driven AND defines a custom structure tree that doesn't mirror US/AC, the agent gets contradictory instructions. No error, no warning. It just picks one.

Scope hints are strategy-agnostic. happy_case / corner_case in rules make sense for scenario-based (EP/BVA naturally separates these). For spec-driven, they're noise — the spec's Given/When/Then already defines what's happy vs. edge. For component-testing, the relevant split is by state lens (empty/loading/error/populated), not happy/corner. The workflow applies scope hints uniformly regardless of strategy.

Dedup creates false orthogonality. The merged-cell dedup rule (write.md Step 3) blanks Level 1/2 when same as previous row. This is a presentation trick, not semantic dedup. Two tests under the same AC with identical steps but different expected results look like they're "related" but may test completely independent behaviors. The dedup masks this — a reviewer sees blank Level 2 and assumes continuity.


2. ATOMICITY — Granularity: MEDIUM-HIGH RISK

Multi-assertion rows. Expected Result allows multiple "SHOULD" bullets per row. A row verifying 5 expectations is 5 tests pretending to be 1. If expectation #3 fails, the case is "failed" — but expectations #4 and #5 are never evaluated. Bug isolation: destroyed.

Real example from the inbox-feb-12 spec — AC1 of US3: Status changes to "Read" + Unread Count decrements + Message A remains visible. That's 3 independent assertions. Current workflow packs them into one Expected Result cell. A CSS bug hiding Message A passes the first two checks but fails the third — yet a tester marks the whole case as failed with no granularity on which assertion broke.

Preconditions are prose, not references. Two tests sharing "I am on the main chat dashboard in the All Chats view" repeat this as text in Steps. No precondition ID or reference system. When the precondition changes (e.g., dashboard gets renamed), you grep-and-replace across all CSV cells — fragile.


3. PARAMETERIZATION — ABSENT

Zero data-driven support. The template has no Test Data or Parameters column. All inputs are hard-coded in Steps prose: "I have a total of 10 messages (7 Read, 3 Unread)", "I click on an unread message (Message A)". To test with 0 messages, 1 message, 100 messages, 10000 messages — you create 4 separate rows with near-identical Steps differing only in the number. No parameterization. No data table. No variable substitution.

CSV is structurally incapable. CSV can't express "run this test with these 5 data sets." You'd need either: a Parameters column with {messageCount} placeholders + a separate data table, or a format that supports parameterized templates (JSON, YAML). The current architecture treats every test case as a unique snowflake. At scale (50+ features, 20+ tests each = 1000+ rows), maintaining data variations is a manual nightmare.


4. MAINTENANCE VELOCITY — SLOW AT SCALE

No cascade propagation. Change a component name in the design system → must update: (1) config.md components list, (2) per-feature context-digest.md, (3) every generated CSV referencing that component name in Steps/Expected Result, (4) feature knowledge files. No automated rename. No search-and-replace tooling. Pure manual grep.

Digest version vs. freshness: logic gap. checkDigestFreshness() only checks file mtimes. If you change the digest generation LOGIC (as we just did with B1/B2 condensing), all existing digests are semantically stale — but the freshness check says "FRESH" because no source files changed. The digest-version: 2 field exists but is never checked by the freshness API. It's decorative.

Clone-on-read creates invisible divergence. getFeatureRules(name) clones global rules on first access. After that, global and per-feature live independently. No diff tooling. No "show which features have customized rules vs stale clones." At 20 features, you have 20 rules files and no idea which ones are intentionally customized.

Template column migration: nonexistent. Add a column to global template → per-feature clones still have old column count. Existing CSVs have wrong column count. No migration path, no warning.


5. LOGIC GAPS — Blind Spots

No state machine model (CRITICAL). The workflow has no first-class concept of state transitions. For inbox-feb-12, the Unread→Read transition, sticky view state, and count decrement are all state-dependent. The agent freestyles these as prose in Steps. No transition matrix, no coverage guarantee.

No negative test category (HIGH). Rules define "Happy Case" and "Corner Case." Negative testing (invalid actions, unauthorized access, destructive operations) is lumped under "Corner Case." But corner is not negative. Empty inbox is a corner case. Deleting someone else's message is a negative case. Different intent, different severity.

No cross-feature interaction (HIGH). Each feature is a silo. No way to express "test inbox filter + notification badge + admin mode simultaneously." State leakage between features is a top-5 bug category in web apps.

Silent knowledge failure (MEDIUM). try/catch with skip on knowledge reads. If knowledge file is malformed (not missing, just bad YAML frontmatter), the digest generates without it, gets marked FRESH, and all subsequent runs use the incomplete digest. No warning in the digest itself.

PDF import hallucination (MEDIUM). pdf-parse is known to garble tables and multi-column layouts. The AI structuring step receives garbage text and must produce clean Markdown. No validation that the structured output actually matches the PDF content. No human review gate between PDF parse and spec save.

Update concurrency (LOW for single user). Update skill reads CSV → modifies in memory → writes back. If the user runs two updates, last-write-wins. Low risk for single-user but blocks future multi-user.


STRUCTURAL REFACTORING
======================

Current model: Monolithic CSV per feature.
Feature → [digest + strategy + rules + template] → Agent → flat CSV

Proposed model: Layered test model.

Feature folder contains:
- test-model.json (structured test definition, not CSV)
  - preconditions[] (reusable, referenced by ID)
  - testSuites[] (grouped by Level 1)
    - testGroups[] (grouped by Level 2)
      - testCases[] (atomic: 1 assertion per case)
        - preconditionRef (ID reference)
        - steps[]
        - assertion (single SHOULD)
        - dataVariants[] (parameterized inputs)
        - tags[] (happy/corner/negative/state)
      - stateTransitions[] (From → Action → To)
- exports/
  - feature-testcase.csv (rendered from model, presentation layer)
  - feature-testcase.json (machine-readable for automation)

Key changes:
- Preconditions as first-class objects with IDs, not embedded prose.
- Single-assertion cases — one SHOULD per case. Multiple SHOULDs = multiple cases.
- Data variants — parameterized inputs per case, rendered as separate rows on export.
- Tags — explicit happy/corner/negative/state categorization.
- State transitions — explicit From→Action→To model, auto-generates transition coverage matrix.
- CSV is a view, not the source of truth — generated from the structured model.

Incremental version: keep CSV as output but add a Parameters column and a Tag column to the template. Enforce single-assertion via a rule constraint.


ADVERSARIAL SCENARIOS
=====================

Scenario 1: State Machine with Branching Rollback

A document approval workflow: Draft → Under Review → Approved → Published → Archived. Rejection paths: Under Review → Draft (with revision notes), Published → Archived (by admin only). Edge: Reviewer rejects, author re-submits, a DIFFERENT reviewer approves, original reviewer tries to re-reject.

Why the workflow struggles: No state transition model. The spec-driven strategy generates one test per AC, but the full state graph has 5 states x 12 transitions = 60 possible paths. The agent will cover the happy path (Draft→Review→Approved→Published) and maybe 2-3 rejection paths. The branching rollback scenario (reject → re-submit → different reviewer) requires tracking entity state across 4 steps with role switching. The write.md Steps column can't express "Actor A does X, then Actor B does Y" cleanly — it assumes a single user journey.


Scenario 2: Combinatorial Filter with Pagination Boundary

A search page with 3 independent filters: date range (7d/30d/90d/custom), status (active/archived/all), type (text/image/video/all). Each combination returns different result counts. Pagination at 25/page. Edge: Filter combination returns exactly 25 results (boundary between 1-page and 2-page). Another edge: Change filter while on page 3 — should reset to page 1.

Why the workflow struggles: scenario-based strategy knows EP/BVA but has no combinatorial awareness. It generates 3+3+4 = 10 partition tests instead of the 3x3x4 = 36 combinations. The pagination boundary (exactly 25 results) is a compound condition — it depends on which filters are active, not on pagination alone. The template has no way to express "this test requires a specific database state" as a precondition.


Scenario 3: Real-Time Component with Race Condition

A chat inbox where: (1) new messages arrive via WebSocket, (2) user is currently viewing the Unread filter, (3) a new unread message arrives while user is reading Message A (sticky view active). Expected: new message appears in list, count increments, but Message A's sticky position is preserved. Edge: 5 messages arrive simultaneously while user switches from Unread to All Chats mid-transition.

Why the workflow struggles: The Steps column assumes synchronous, user-initiated actions. There's no concept of "while the user is doing X, the system does Y concurrently." The verb whitelist (NAVIGATE/CLICK/INPUT/SELECT/VERIFY/HOVER) has no verb for "WAIT FOR" or "OBSERVE" or "DURING." The Expected Result can say "SHOULD display new message" but can't express timing — when should it appear relative to the user's current action? The workflow has no temporal dimension.
